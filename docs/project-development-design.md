# AI Resume Assistant 项目开发设计文档

## 1. 文档目的

本文档基于当前仓库 `ai-resume-assistant` 的实际实现整理，目标是为项目后续开发、重构、扩展和交付提供统一参考。  
文档既描述当前版本已经落地的方案，也给出适合下一阶段演进的设计建议，避免后续开发只停留在“能跑起来”，而缺少接口规范、异常处理和稳定性约束。


## 2. 项目概述

### 2.1 项目名称

AI 简历对话工作台

### 2.2 项目目标

为求职用户提供一套围绕简历展开的智能辅助工具，支持：

- 上传并解析简历文件
- 基于简历内容进行多轮问答
- 结合岗位 JD 进行匹配分析
- 输出匹配度、缺失项和优化建议

### 2.3 核心价值

- 把“静态简历”转化为“可交互的求职上下文”
- 把“泛泛提问”变成“围绕简历内容的连续追问”
- 把“主观优化建议”结构化为可执行的 JD 匹配结果

### 2.4 当前版本定位

当前项目更接近一个可演示、可继续演进的 V1 原型，已经具备前后端联调能力和核心业务闭环，但在接口标准化、日志监控、限流、测试覆盖、可维护性拆分等方面仍有继续完善空间。

## 3. 业务范围与用户流程

### 3.1 目标用户

- 准备实习或校招的学生
- 需要优化简历表达的求职者
- 想快速判断某个 JD 是否匹配自己经历的用户

### 3.2 核心业务流程

1. 用户上传 PDF、DOCX 或 TXT 简历
2. 后端解析出简历纯文本
3. 前端保存简历文本作为后续问答上下文
4. 用户围绕简历进行多轮提问
5. 后端调用大模型生成基于简历的回答
6. 用户输入岗位 JD
7. 后端调用大模型返回结构化匹配结果
8. 前端展示匹配度、匹配项、缺失项和建议

### 3.3 非目标范围

当前版本暂不包含：

- 用户账号体系
- 简历长期云端存储
- 多份简历管理
- 管理后台
- 支付系统
- 企业端职位管理

## 4. 技术选型

### 4.1 前端

- React 19
- TypeScript
- Vite
- Axios
- CSS 原生样式

选择原因：

- React 适合构建交互式单页应用
- TypeScript 有利于接口约束和状态管理
- Vite 启动和构建速度快，适合前期快速迭代
- Axios 简化 HTTP 调用和异常处理

### 4.2 后端

- FastAPI
- Pydantic
- Uvicorn
- python-docx
- pdfplumber
- OpenAI Python SDK
- python-dotenv

选择原因：

- FastAPI 对接口开发、参数校验和文档生成非常友好
- Pydantic 能提升请求体和响应体结构化程度
- 文档解析库满足 PDF、DOCX、TXT 三类简历输入
- OpenAI SDK 便于接入兼容 OpenAI 协议的大模型平台

## 6. 总体架构设计

### 6.1 架构模式

当前项目采用典型的前后端分离架构：

- 前端负责页面展示、状态管理、交互控制
- 后端负责文件解析、业务编排、模型调用
- 大模型服务作为外部能力提供方

### 6.2 架构关系

```text
用户浏览器
   ↓
React 前端
   ↓ HTTP
FastAPI 后端
   ├─ 文件解析服务
   ├─ Prompt 构造服务
   └─ LLM 调用服务
        ↓
     大模型平台
```

### 6.3 当前架构优点

- 简单直接，联调成本低
- 前后端职责边界较清晰
- 核心链路短，适合课程项目和原型验证

### 6.4 当前架构不足

- 前端逻辑集中在单一组件，后续维护成本会升高
- 后端没有独立 schema、config、exception、logging 模块
- 没有统一响应结构和错误码规范
- 没有缓存、限流、监控等工程化能力

## 8. 核心接口设计

当前项目有三个核心接口。

## 8.1 简历上传接口

### 接口定义

- 方法：`POST`
- 路径：`/api/resume/upload`
- Content-Type：`multipart/form-data`

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `file` | File | 是 | 简历文件，支持 `.pdf` `.docx` `.txt` |

### 成功响应

```json
{
  "success": true,
  "fileName": "resume.docx",
  "resumeText": "解析后的简历文本"
}
```

### 失败响应

当前实现失败时通过 `HTTPException` 返回，例如：

```json
{
  "detail": "简历解析失败：具体错误"
}
```

### 设计说明

- 上传文件先临时落盘
- 解析完成后立即删除临时文件
- 使用随机文件名，避免同名覆盖

### 风险点

- 大文件上传未限制大小
- 只支持文字版 PDF，扫描件 PDF 仍可能提取失败
- 缺少 MIME 类型与扩展名双重校验

## 8.2 简历问答接口

### 接口定义

- 方法：`POST`
- 路径：`/api/chat/ask`
- Content-Type：`application/json`

### 请求体

```json
{
  "resumeText": "简历内容",
  "question": "请介绍一下这份简历适合哪些岗位",
  "messages": [
    {
      "role": "user",
      "content": "这份简历适合哪些岗位"
    },
    {
      "role": "assistant",
      "content": "这份简历更适合..."
    }
  ]
}
```

### 成功响应

```json
{
  "success": true,
  "answer": "模型输出内容",
  "source": "llm"
}
```

### 兜底响应

当模型调用失败时，当前实现会返回：

```json
{
  "success": true,
  "answer": "兜底提示文本",
  "source": "mock"
}
```

### 设计说明

- 优先从 `messages` 中获取最后一条用户问题
- 支持多轮上下文问答
- 对简历文本和问题长度做截断，避免 prompt 过长
- 前端收到完整文本后，以打字机效果逐字显示

### 风险点

- 当前把模型失败也包装成 `success: true`，会弱化真实错误感知
- 缺少请求 ID 和日志链路，定位线上问题会较难
- 当前仍是“一次性返回”，不是真正流式输出

## 8.3 JD 匹配接口

### 接口定义

- 方法：`POST`
- 路径：`/api/jd/match`
- Content-Type：`application/json`

### 请求体

```json
{
  "resumeText": "简历内容",
  "jdText": "岗位 JD 内容"
}
```

### 成功响应

```json
{
  "success": true,
  "result": {
    "score": 82,
    "matched": ["具备后端开发经验", "有项目实践", "掌握 Python"],
    "missing": ["缺少量化成果", "缺少特定业务经验", "缺少云平台经验"],
    "suggestions": ["补充关键词", "量化项目结果", "针对岗位调整简历"]
  },
  "source": "llm"
}
```

### 兜底响应

模型调用失败或 JSON 解析失败时，当前会返回默认结构：

```json
{
  "success": true,
  "result": {
    "score": 72,
    "matched": ["..."],
    "missing": ["..."],
    "suggestions": ["..."]
  },
  "source": "mock",
  "message": "大模型分析失败，已返回兜底结果"
}
```

### 设计说明

- 后端要求模型直接输出 JSON 文本
- 服务层再反序列化成对象后返回给前端
- 前端据此渲染统计卡片和结果列表

### 风险点

- 依赖模型严格输出 JSON，格式稳定性有限
- 没有对字段做二次结构校验
- 兜底结果固定，可能让用户误以为是真实分析结果

## 9. 数据与状态设计

## 9.1 前端状态

当前前端核心状态包括：

- `file`：当前选择的简历文件
- `resumeText`：解析后的简历文本
- `sessions`：所有聊天会话
- `activeSessionId`：当前激活会话
- `question`：输入中的问题
- `typingState`：打字机显示状态
- `jdText`：岗位 JD 文本
- `matchResult`：JD 匹配结果
- `toast`：页面轻提示信息

### 当前设计优点

- 业务状态完整
- 与页面交互一一对应
- 会话支持持久化到 `localStorage`

### 当前设计不足

- 状态都集中在一个组件中
- 后续功能继续增长时会让组件越来越难维护

## 9.2 后端数据流

### 上传解析链路

`UploadFile -> 临时文件 -> parse_resume -> 文本返回 -> 删除临时文件`

### 问答链路

`resumeText + messages -> normalize_chat_messages -> build prompt -> LLM -> answer`

### JD 链路

`resumeText + jdText -> build_jd_match_prompt -> LLM -> JSON parse -> result`

## 10. 文件解析设计

## 10.1 支持类型

- TXT
- PDF
- DOCX

## 10.2 当前实现策略

### TXT

- 直接按 UTF-8 读取文本

### PDF

- 使用 `pdfplumber` 分页提取文本

### DOCX

- 优先使用 `python-docx` 读取段落
- 如果段落为空，则回退到直接扫描 `word/*.xml` 中的 `w:t` 节点

### 文本清洗

- 对连续重复行做去重
- 返回拼接后的纯文本内容

## 10.3 当前优势

- 对常见简历格式支持完整
- DOCX 针对文本框模板做了兼容增强
- 临时文件会及时清理

## 10.4 仍需完善的点

- 扫描版 PDF 无 OCR 能力
- 未做文件大小限制
- 未对异常文件或恶意文件做隔离校验
- 缺少解析质量评分，例如“是否提取到有效内容”

## 11. Prompt 与模型调用设计

## 11.1 当前实现

当前后端将 Prompt 生成和模型调用分为两层：

- `prompt_builder.py`：负责生成 Prompt
- `llm_service.py`：负责模型调用、参数设置和内容截断

## 11.2 当前已做的控制

- 对简历内容和问题长度做裁剪
- 多轮聊天只保留最近若干条消息
- 要求模型使用中文回答
- 要求 JD 接口返回 JSON
- 限制模型不要编造简历中不存在的信息

## 11.3 潜在问题

- Prompt 仍以字符串模板为主，后续维护容易散
- 不同模型平台对 JSON 输出稳定性不同
- 没有模型级重试、超时细分和降级策略

## 14. 性能设计

虽然项目规模不大，但仍建议尽早考虑性能边界。

## 14.1 当前性能措施

- 简历文本和问题文本做长度截断
- 前端搜索使用 `useDeferredValue`
- 本地会话数据使用浏览器缓存，减少后端依赖
- 打字效果在前端实现，不增加后端流式复杂度

## 14.2 性能风险

- `App.tsx` 状态集中，未来渲染负担会增加
- 会话过多时，`localStorage` 和列表渲染会变慢
- 模型调用是主要耗时瓶颈
- 大文件解析和超长 JD 会影响响应时间

## 14.3 优化建议

- 聊天会话列表增加分页或懒加载
- 对 JD 分析和问答接口增加超时提示
- 对模型接口增加缓存策略，例如相同 JD 的重复分析缓存
- 前端组件化拆分，减少不必要重渲染
- 长文本解析改为异步任务时，可引入任务队列

---

## 15. 稳定性设计

## 15.1 关键稳定性风险

- 模型接口超时或不可用
- DOCX/PDF 解析失败
- 前后端跨域或环境变量配置错误
- 大模型输出非法 JSON
- 上传文件内容异常导致流程中断

## 15.2 当前已有保护

- 模型请求设置了超时
- 对模型失败有兜底文本或兜底结构
- 上传后临时文件会自动清理
- 输入内容在服务层有长度裁剪

## 15.3 建议增强

- 为每次请求生成 `requestId`
- 增加接口访问日志和异常日志
- 模型调用加入重试和降级策略
- JD 输出增加严格 schema 校验
- 限制上传文件大小和解析时长
- 对上传解析增加异步任务状态，如“排队中/解析中/完成/失败”

---

## 16. 安全设计

## 16.1 当前需要关注的问题

- CORS 当前为 `allow_origins=["*"]`，仅适合开发阶段
- 上传文件缺少更严格校验
- 简历内容属于敏感个人信息
- 前端当前直接保存会话到本地浏览器，存在隐私风险

## 16.2 建议方案

- 生产环境限制允许访问的域名
- 校验文件扩展名、MIME 类型和大小
- 对上传内容增加敏感信息提示
- 明确隐私声明，说明简历内容是否会发送到第三方模型平台
- 若面向真实用户，考虑对会话数据进行加密存储或后端托管

## 17. 日志、监控与运维建议

当前项目尚未引入系统化日志与监控，但这是上线前必须补的能力。

### 17.1 建议记录的日志

- 请求路径、方法、状态码、耗时
- 上传文件名、类型、大小
- 模型调用开始时间、结束时间、耗时
- 模型平台错误码或异常信息
- 解析失败文件类型和失败原因

### 17.2 建议监控指标

- 简历上传成功率
- 文档解析成功率
- 聊天接口平均耗时
- JD 匹配接口平均耗时
- 模型调用失败率
- JSON 解析失败率

### 17.3 建议告警条件

- 模型失败率在 5 分钟内超过阈值
- 文档解析失败率突然升高
- 接口平均耗时显著增加
